Basic theory

Based on wikipedia

  • Types:
    • Static (reviews, inspection, etc.)
    • Dynamic (execution of program againt test case)
  • Approaches:
    • Black box (examining functionality without any knowledge of internal implementation)
    • White box(tests internal structures or workings of a program)
    • Gray box (executing tests at the user, or black-box level, with having knowledge of internal data structures and algorithms for purposes of designing tests)
  • Levels:
    • Unit (component testing)
    • Integration (testing interfaces between components against a software design)
    • System(tests a completely integrated system to verify that it meets its requirement)
    • Acceptance

Objectives of testing

Based on wikipedia

  • Installation (verify installation)
  • Compatibility (testing compatibility with other SW - e.g. OS)
  • Sanity (determines whether it is reasonable to proceed with further testing)
  • Smoke (determines whether there are serious problems)
  • Regression (testing focused on finding defects after a major code change has occurred)
  • Functional (testing to verify a specific action or function of the code meets requirement or specification)
  • Performance (determines how a system or sub-system performs in terms of responsiveness and stability under a particular workload)
  • Security (determines prepareness of system to prevent system intrusion by hackers)

Mocks & Stubs - comparison

Based on spring training materials

Advantages Disadvantages
Mock
  • No additional class to maintain
  • Only relevant for the test
  • Test behavior and state
  • A little harder to understand (at first)
Stub
  • Easy to implement and understand
  • Reusable
  • Needs update when the interface is changed
  • Must implement all methods
  • Refactoring can break other test (where the stub is reused)

Mocks & Stubs - general recommendations

Based on spring training materials

  • Use mocks for non-trivial interfaces
  • Use stubs for simple interfaces with repeated functionality
  • Always consider the specific situation

Rules for writting tests

  • independency
  • performance
  • only in test package
  • high test coverage != good tests
